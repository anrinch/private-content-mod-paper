\section{Blocklist Constructions}
%
In this section, we will discuss constructions for authenticated and confidential blocklists (ACB) leveraging new cryptographic building blocks. To securely realize the \checkContent and \checkContentFuzzy interfaces in the ideal functionality, we will ensure that an ACB supports \textbf{oblivious matching}: the client can obliviously match its input against items in the blocklist and reveal the result to the server; If the client's input is not in the blocklist, then the result of the match does not reveal any information.

\myparagraph{Content moderation from authenticated blocklists}
%
A secure ACB construction almost immediately provides a secure realization of \contentModFuncExact. We outline the steps below

\enumListStart
\item The auditors invoke \publishList with the blocklisted items \blocklist, which outputs $\blindedBlocklist$ and the signatures under the auditors' private keys. The client downloads \blindedBlocklist and verifies the signatures. 
\item If the ACB supports preprocessing, then the client and server invoke \preprocess to produce $\blklistToken$ that is stored on the client. If there is no preprocessing, then $\blklistToken = \blindedBlocklist$. 
\item When a client wants to upload content, the client and server invoke \searchBlocklist which enables the server to determine if the client's content is on the blocklist. 
\enumListEnd

We will introduce three different types of authenticated blocklists, and use these constructions later in \secref{} to design our content moderation schemes. 



\subsection{Bloom Filter Based Blocklist}
%
A Bloom filter provides a succinct representation of a set with storage costs that are sublinear in the set size. A Bloom filter is ideally suited for a large blocklist since it is stored on the client. However, a Bloom filter does not provide confidentiality of the blocklisted items, as required by our functionality\footnote{We note that a Bloom filter without confidentiality was considered in prior work but assumed to be unsatisfactory.}. 


Our key idea to introduce confidentiality is to construct the Bloom filter with items that are obfuscated against the client but still enables efficient searches. This is supported 




\subsubsection{Authenticated OPRF}
%
We define a functionality called \emph{authenticated OPRF}, that extends the definition of an OPRF to a three party functionality \authOPRFFunc (\figref{fig:authOPRF_func}), where a group of $\numAuditors$ auditors with semi-honest majority jointly sign a list of inputs to a pseudorandom function. \authOPRFFunc receives the key for the PRF from the client, who acts as the sender. The receiver of the PRF evaluations is the server, who does not input anything.

\myparagraph{Realizing \authOPRFFunc}
%
We realize \authOPRFFunc by leveraging the the 2HashDH PRF construction~\cite{jarecki2014round} in a group where the OM gap-DH problem is hard. The construction is $\prf{\prfKeyClient}(\genericFnInput) = \oprfRO[2](\genericFnInput, \oprfRO[1](\genericFnInput)^{\prfKeyClient})$, where $\oprfRO[1], \oprfRO[2]$ are modeled as random oracles. The full protocol is in \figref{fig:authOPRF} in \appref{app:authOPRF}. In brief,

\enumListStart
\item The auditors ``blind'' the blocklist elements $\blindedBlocklist \assign \{ \oprfRO[1](\genericBlocklistElmt[\setIdx])^{\oprfBlindingFactor[\setIdx]}\}_{\genericBlocklistElmt[\setIdx] \in \blocklist}$ and the server is provided the blinding factors $\oprfBlindingFactor[1], \dots, \oprfBlindingFactor[\blocklistSize]$ . 
\item The auditors publish \blindedBlocklist and sign (a hash of) \blindedBlocklist under their private keys. The client obtains \blindedBlocklist and the auditors' signatures. 
\item After verifying the signatures, the client exponentiates the blinded elements $\{ \left(\oprfRO[1](\genericBlocklistElmt[\setIdx])^{\oprfBlindingFactor[\setIdx]}\right)^{\prfKeyClient}\}_{\genericBlocklistElmt[\setIdx] \in \blocklist}$ and sends them to the server. 
\item The server removes the blinding and computes $\{ \oprfRO[2](\genericBlocklistElmt[\setIdx], \oprfRO[1](\genericBlocklistElmt[\setIdx])^{\prfKeyClient}\}_{\genericBlocklistElmt[\setIdx] \in \blocklist}$. 
\enumListEnd

\subsection{Homomorphic Signature Based Blocklist}
%
There are two drawbacks of the Bloom filter based ACB construction: i) the storage overhead for \blklistToken is $\bigO(\blocklistSize)$, and ii) the search can produce false positives. In this section, we will design a ACB that has zero false positives and false negatives and constant storage overhead, but with higher computation costs during search. The construction leverages a new cryptographic primitive called a \emph{randomizable set homomorphic signature}. Intuitively, the ACB is constructed from a \emph{set-homomorphic} signature on the accumulated set of blocklisted items~\cite{johnson2002homomorphic} under the auditors' private keys. The signature scheme has homomorphic properties that allows us to compute signatures of subsets of a signed set, as well as the union of two signed sets. We leverage these homomorphic properties to directly use the signature as the ACB and compute on the signature to implement an oblivious match. Importantly, the ACB in this construction is constant-sized making it highly storage efficient.

\begin{remark}

We denote this signature with $\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]]$. 
 However, when $\genericCInputElmt \notin \blocklist$, $\genericSig[\blocklist] \sigMinus[\pubKey] \{\genericCInputElmt\}$ is an element in the output space $\sigOutputSpace$ that is not necessarily equal to $\genericSig[\blocklist]$. As will be evident later, 
$\genericSig[\blocklist] \sigMinus[\pubKey] \genericCInputElmt$ is a signature on an unpredictable element in \sigOutputSpace, and with abuse of notation we also denote this element as $\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]]$. 
\end{remark}

A randomizable set homomorphic signature scheme must satisfy two security properties 

\itemListStart

$\genericCInputElmt$. We formalize this property as \emph{indistinguishable signatures under randomizations (IND-SR)}, using a CPA-style definition such that for $\genericCInputElmt{0}, \genericCInputElmt{1} \notin \blocklist$,

\myparagraph{Correctness}
%
The correctness of $\keyGen$, $\signingAlg$ and $\verify$ is obvious. We describe how the signature scheme satisfies the homomorphic properties. All calculation are performed modulo $\rsaMod$; we omit this factor from the description for notational simplicity. 

Let  $\genericSetVar[1]$, $\genericSetVar[2]$ be two sets and $\accSet[1] \gets 
\prod\limits_{\genericCInputElmt \in \genericSetVar[1]} \primeRO(\genericCInputElmt)$, and $\accSet[2] \gets 
\prod\limits_{\genericCInputElmt' \in \genericSetVar[2]}  \primeRO(\genericCInputElmt')$. Also, let the signatures on $\genericSetVar[1]$ and $\genericSetVar[2]$ be $\genericSig[\genericSetVar[1]] \assign \qrGen^{1/\accSet[1]}$ and $\genericSig[\genericSetVar[2]] \assign \qrGen^{1/\accSet[2]}$ respectively. %Now, note that


\itemListStart
\item \itemHeading{Union of sets:} We define the homomorphic operator $\sigHomomorphism$ below


\begin{align*}
& \genericSig[\genericSetVar[1]] \sigHomomorphism[\pubKey] \genericSig[\genericSetVar[2]] = \genericSig[\genericSetVar[1] \cup \genericSetVar[2]] = 
\genericSig[\genericSetVar[1]]^{\euclideanConst[2]}  \genericSig[\genericSetVar[2]]^{\euclideanConst[1]} = \qrGen^{\euclideanConst[2]/\accSet[1]} \qrGen^{\euclideanConst[1]/\accSet[2]} \\
&  = \qrGen^{\frac{\euclideanConst[2] \accSet[2] + \euclideanConst[1] \accSet[1]}{\accSet[1] \accSet[2]}} =\qrGen^{\frac{\gcd(\accSet[1], \accSet[2])}{\accSet[1] \accSet[2]}} = \qrGen^{1/\lcm(\accSet[1], \accSet[2])} \label{eq}
\end{align*}


Here, $\euclideanConst[1], \euclideanConst[2]$ are integers such that $\euclideanConst[1]\accSet[1] + \euclideanConst[2] \accSet[2] = \gcd(\accSet[1], \accSet[2])$, and they can be found in polynomial time using the extended Euclid algorithm. Due to Bezout's identity $\euclideanConst[1], \euclideanConst[2]$ always exist. 






\item \itemHeading{Difference of two sets:} We define the set difference operator $\sigMinus$ below

\[
\genericSig[\genericSetVar[1]] \sigMinus[\pubKey] \genericSetVar[2] = \genericSig[\genericSetVar[1] \setminus \genericSetVar[2]] = \genericSig[\genericSetVar[1]]^{\accSet[2]} = \qrGen^{\accSet[2]/\accSet[1]}
\]

, where $\genericSetVar[2] \subseteq \genericSetVar[1]$, 


\item \itemHeading{Homomorphic property over randomization:} The set difference operator $\sigMinus$ is homomorphic over the randomization function.    


\begin{align*} 
& \randomizeFn[\pubKey](\genericSig[\genericSetVar[1]], \genericRand) 
\sigMinus[\pubKey] \genericSetVar[2] = \qrGen^{\genericRand \fieldMult \accSet[2]/\accSet[1]}  = \\
&\randomizeFn[\pubKey]\left(\qrGen^{\accSet[2]/\accSet[1]}, \genericRand\right) = \randomizeFn[\pubKey](\genericSig[\genericSetVar[1]] \sigMinus[\pubKey] \genericSetVar[2], \genericRand)
\end{align*}

\itemListEnd


We show in \appref{sec:proof:sig_scheme} that the security of the scheme reduces to the 
strong QR-RSA assumption in the random oracle model.

\subsection{Encrypted Cuckoo Hash Based Blocklist}
%
The blocklist constructions we have considered so far do not support threshold searching. In this section, we will design a ACB that supports threshold search. The construction leverages a cuckoo hash table. Intuitively, the blocklisted items are hashed into a cuckoo hash table \cuckooTable of size $\blocklistSize \numProjections(1+\cuckooTableParam)$ with hash functions $\cuckooHashFn{1}, \cuckooHashFn{2}$, where $\cuckooTableParam$ is a parameter that controls false negative rate~\cite{bhowmick2021apple}. The hash table is stored on the client and the values in each location is individually encrypted under an additively homomorphic encryption scheme. The hash keys are not stored alongside: the client sees $2 \blocklistSize \numProjections(1+\cuckooTableParam)$ encrypted values, and knows the relation between ciphertexts and their corresponding locations in the table. The empty locations in the table are padded with encryptions of dummy values. 


During a threshold search the client performs lookups in \cuckooTable, and then compares its inputs with the encrypted values in the corresponding locations in the hash table, by homomorphically subtracting one from the other, and blinding the result with a random multiplicative factor. If the projections match, then after decryption, the server should observe an encryption of \groupIdentity. 
To support threshold searches, the client secret-shares a known value in the encrypted results provided to the server. After decryption, if the server obtains enough shares, then it can reconstruct the value and confirm. The main challenge is

\subsubsection{Detectable Secret Sharing}
%
Similar to standard $(\secretShareLength, \secretShareThreshold)$ secret-sharing, a detectable secret sharing (DSS) scheme has two algorithm $\detectableSecretSharingScheme = \langle \DSShare, \DSCombine\rangle$. 

\itemListStart
\item $\{ \secretShare{1},\dots, \secretShare{\secretShareLength} \} \gets \DSShare(\genericSymmKey)$ takes as input a secret $\genericSymmKey$ and it produces a set of secret-shares $\{ \secretShare{1},\dots, \secretShare{\secretShareLength} \}$
\item $\genericSymmKey/\bot \gets \dssProtocol.\DSCombine(\{\genericSetElmt[1], \dots, \genericSetElmt[\secretShareLength] \})$ Given a set of $\secretShareLength$ elements, if at least \secretShareThreshold of them are valid secret shares of the same secret, and the rest are random elements, output the secret $\genericSymmKey$. Otherwise, output $\bot$. 
\itemListEnd

For security \dssProtocol ensures that when there are less that \secretShareThreshold number of shares in $\{\genericSetElmt[1], \dots, \genericSetElmt[\secretShareLength] \}$ then nothing is revealed to an adversary that has the set \emph{including which of the shares are correct}. 



\myparagraph{Detectable hash function}
%
We will make use of a primitive introduced by \citet{bhowmick2021apple} called a detectable hash 
function to realize a DSS scheme. Formally, a $(\dhfParamMatchThreshold, \dhfParamRandThreshold)$-detectable 
hash function $\dhf: \dhfKeySpace \times \dhfDomain \rightarrow \dhfRange$ has the following 
two properties.

\enumListStart
\item The function is weakly \textit{\dhfParamMatchThreshold-wise
  independent}, that is the distribution of
  $\langle\dhf[\dhfKey](\genericFnInput[1]), \dots,
  \dhf[\dhfKey](\genericFnInput[\dhfParamMatchThreshold])\rangle$
  given $\dhfKey \getsr \dhfKeySpace$ and $\genericFnInput[1], \dots,
  \genericFnInput[\dhfParamMatchThreshold] \getsr \dhfDomain$ is
  identical to the uniform distribution over
  $\dhfDomain^{\dhfParamMatchThreshold}$.
	
\item The function is \textit{\dhfParamRandThreshold-detectable}. 
  Given a vector $\genericVec = \tupleDefn{\genericVec[1], \dots, 
    \genericVec[\dhfLength]} \in \dhfDomain^{\dhfLength}$, where for $\matchedIndexSet 
  \subseteq \nats[\dhfLength]$, $\genericVec[\projIdx] \gets 
  \dhf[\dhfKey](\genericFnInput[\projIdx])$ when $\projIdx \in \matchedIndexSet$, 
  and $\genericVec[\projIdx] \getsr \dhfDomain$ when $\projIdx \notin \matchedIndexSet$, 
  there is an algorithm $\dhfDetect$   
	\itemListStart
  \item $\dhf[\dhfKey].\dhfDetect(\genericVec) =
    \matchedIndexSet$ with non-negligible probability,
    when $\setSize{\matchedIndexSet} \ge
    \max(\dhfParamMatchThreshold+1, \genericVecLen -
    \dhfParamRandThreshold)$.
  \item $\dhf[\dhfKey].\dhfDetect(\genericVec) =
    \dhfFail$ when $\setSize{\matchedIndexSet} \le
    \dhfParamMatchThreshold$.
\itemListEnd
\enumListEnd
  

  Intuitively, given a vector $\genericVec \in
  \dhfDomain^{\dhfLength}$, where at least
  $\max(\dhfParamMatchThreshold+1, \dhfLength - \dhfParamRandThreshold)$ 
  components are outputs of $\dhf$,
  the detection algorithm identifies these components. However,
  if less than $\dhfParamMatchThreshold+1$ entries are generated
  by $\dhf$, then the detection algorithm does not reveal any
  information about $\matchedIndexSet$.		


\myparagraph{Protocol for detectable secret sharing}
The protocol for a DSS is as follows. 

\begin{algorithm}
\caption{\detectableSecretSharingScheme: detectable secret sharing scheme}
\footnotesize
\begin{algorithmic}[1]

\LComment{Secret sharing a value $\genericSymmKey$ into $\secretShareLength$ shares}
\Procedure{Share}{$\genericSymmKey$}   
\State Secret-shares \genericSymmKey into \secretShareLength shares using a $(\secretShareLength, \secretShareThreshold)$ secret sharing scheme $\secretSharingScheme \assign \langle \SSShare, \SSCombine\rangle$: $\{ \secretShare{1}, \dots, \secretShare{\secretShareLength} \} \assign \SSShare(\genericSymmKey)$
\State The dealer samples $\secretShareLength$ one-time pads, $\genericSymmKey_1, \dots, \genericSymmKey_{\secretShareLength}$ and computes the set $\ssSet \assign \{\langle \genericSymmKey_1, \secretShare{1}, \rangle, \dots, \langle \genericSymmKey_n, \secretShare{\secretShareLength} \rangle \}$
\State The dealer samples a key for a $(\secretShareLength - \secretShareThreshold, \secretShareThreshold - 1)$-detectable hash function \dhfKey, and computes the set $$\dhfSet \assign \{\dhf[\dhfKey](\genericRand[1])) \oplus \genericSymmKey_1, \dots, \dhf[\dhfKey](\genericRand[n]))\oplus \genericSymmKey_n\}$$, where $\genericRand[1], \dots, \genericRand[\secretShareLength] \sample \dhfDomain$.
\State \Return \ssSet, \dhfSet
\EndProcedure
\Statex
\LComment{Reconstructing the detectable secret shares }
\Procedure{Reconstruct}{$\ssSet' \assign \{\langle \genericSymmKey_1', \secretShare{1}', \rangle, \dots, \langle \genericSymmKey_{\secretShareLength}', \secretShare{\secretShareLength}' \rangle \}$, $\dhfSet  \}$} 
\State Compute the vector $\genericVec \assign \langle \genericSymmKey_\setIdx' \oplus \dhfSetElmt{\setIdx}\rangle_{\setIdx \in [1, \secretShareLength]}$ 
\State Compute $\dhf[\dhfKey].\dhfDetect(\genericVec) =
    \matchedIndexSet$. If the algorithm fails, then abort. Otherwise, construct $\genericSetVar \assign \{ \secretShare{\setIdx}' :   \setIdx \in \matchedIndexSet \}$ and reconstruct the secret shares $\genericSymmKey \assign \SSCombine(\genericSetVar)$ 
\State \Return $\genericSymmKey$/$\bot$

\EndProcedure
\end{algorithmic}
\end{algorithm}
