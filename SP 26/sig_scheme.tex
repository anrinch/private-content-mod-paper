\myparagraph{Motivation}
%
Recall that we need to design the blocklist authentication token \blklistToken{\blocklist} and content verification token \contentToken{\clientData} such that they satisfy the security requirements discussed in \secref{sec:model}. In addition, since blocklists can be large in size~\cite{content_mod_e2ee},  we want to ensure that: i) the \blklistToken{\blocklist}, that must be stored or downloaded by the client, is sublinear in the blocklist size, and ii) the client can compute the \contentToken{\clientData} with costs that are sublinear in the blocklist size.

%, and iii) the entire process is non-interactive and requires communication that is 
%sublinear in the blocklist size.  

%Unfortunately, existing designs do not optimize all three metrics (see \tblref{tab:related}). 

We propose a novel cryptographic primitive called a \textit{randomizable set homomorphic signature} to address these goals. Intuitively, we will use a signature by the auditors on the blocklist as  \blklistToken{\blocklist} since it is unforgeable. Then  our key idea is to design a scheme that directly allows computation on the signature, in order to compute \contentToken{\clientData}. If the size of the signature scales sublinearly in the blocklist size, and the computation to derive \contentToken{\clientData} scales linearly in the size of the signature (and the client's input), then we achieve a scheme with sublinear compute costs for the client.

In particular, we start with a \emph{set homomorphic signature scheme}~\cite{johnson2002homomorphic}, that allows 
a party (or a group) to sign a set of items such that the size of the signature is sublinear in the size of the set. This is achieved by accumulating the set using a cryptographic accumulator in a hidden-order group~\cite{baric1997collision}, where the order serves as the private key. 
Additionally, the signature is homomorphic, i.e., we can compute signatures on subsets of a signed set, and the union of two 
signed sets, using  homomorphic operators that only require the public key. 

Now, consider the following design. The auditors publish a signature $\genericSig[\blocklist]$ on the blocklist $\blocklist$.

This is possible by using the public key and the homomorphic operator. $\genericSig[\blocklist]$ and $\genericCInputElmt$ requires sublinear compute costs.

\begin{definition}

	collection of five polynomial-time algorithms \ourSigSchemeDefn. The key space is \sigKeySpace and the output space is $\sigOutputSpace$.
	
	\itemListStart
		\item $\keyPair \assign \keyGen(1^{\secParam})$: Generates a public-private key pair 	

		\item $\genericSig[\genericSetVar] \assign \signingAlg[\privKey](\genericSetVar)$: On input a set 
		$\genericSetVar \subseteq 
		\universe$, and private key $\privKey$ generated by $\keyGen$, the algorithm produces a 
		signature $\genericSig \in \sigOutputSpace$.

		\universe$, a signature $\genericSig \gets
		\signingAlg[\privKey](\genericSetVar)$ and the
		public key $\pubKey$ corresponding to $\privKey$,
		the algorithm outputs 1.
		
		\item $\genericSigPairAlt \assign \randomize[\pubKey](\genericSigPair[1])$: On input an arbitrary element from 
		the output space $\genericSigPair[1] \in \sigOutputSpace$, the algorithm outputs

%\myparagraph{Remark on notation}

\begin{remark}
In our discussions we will denote a signature on the set $\genericSetVar \subseteq\universe$ as $\genericSig[\genericSetVar]$ for notational simplicity. For any $\genericCInputElmt \in \universe$, the computation $\genericSig[\blocklist] \sigMinus[\pubKey] \genericCInputElmt$ returns a valid signature on the set $\genericSetDiff[\blocklist][\genericCInputElmt]$ when $\genericCInputElmt \in \blocklist$ due to \eqnref{eq:sig_homomorphism_minus}. We denote this signature with $\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]]$. 
 However, when $\genericCInputElmt \notin \blocklist$, $\genericSig[\blocklist] \sigMinus[\pubKey] \genericCInputElmt$ is an element in the output space $\sigOutputSpace$ that is not necessarily equal to $\genericSig[\blocklist]$. As will be evident later, 
$\genericSig[\blocklist] \sigMinus[\pubKey] \genericCInputElmt$ is a signature on an unpredictable element in \sigOutputSpace, and with abuse of notation we also denote this element as $\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]]$. 
\end{remark}

\begin{remark}

\subsection{Distributed Key Generation \& Signing}
%
In our description so far, we have assumed that there is a single party with the private key. However, to leverage the signature scheme in our protocol, where multiple auditors sign the blocklist, we need to extend it to a distributed setting such that the auditors only have a share of the private key. Additionally, the distributed key generation and sharing must be robust against a malicious minority of signers. Fortunately, our signature scheme can be extended to support distributed key generation and singing using existing techniques. Due to space constraints, we define the ideal functionality \distSigningFunc for distributed key generating and singing, and describe the protocol realizing the functionality in \appref{app:dist_signing}. We provide a high level overview of the solution below. 

\myparagraph{Key generation}
%
Recall that the public key of the signature scheme is the modulus $\rsaMod$ and a generator $\qrGen$, and the private key is the factorization of $\rsaMod$. We use the protocol of \citet{algesheimer2002efficient}\footnote{To the best of our knowledge this is the only protocol that ensures that $\rsaMod$ is generated from safe primes, which is necessary to ensure that $\qrGroup$ is cyclic} to generate a shared modulus $\rsaMod = (2 \sgPrime[1] + 1) (2 \sgPrime[2] + 1)$ such that the auditors only have additive shares of $\sgPrime[1], \sgPrime[2]$.  
 The shares allow the auditors to securely compute $\rsaMod$ and $\orderFn(\qrGroup) = \sgPrime[1] \sgPrime[2]$ using the BGW protocol~\cite{asharov2017full}. \rsaMod is included in the public key while shares of $\orderFn(\qrGroup)$ are distributed to the auditors and kept as the private key. 

The generator selection is also performed using the BGW protocol by checking that for a random value $\qrGen \in [2, \rsaMod -1]$, selected by the auditors, $\qrGen^{\sgPrime[1]} \not \equiv 1 \bmod \rsaMod ~\land ~\qrGen^{\sgPrime[2]} \not \equiv 1 \bmod \rsaMod$. 
It is necessary to perform this check without revealing the outputs of   
$\qrGen^{\sgPrime[1]}, \qrGen^{\sgPrime[2]}$ in the clear, and thus we use the BGW instantiations. We introduce several optimizations to reduce the number of BGW invocations by trivially excluding cases where $\qrGen \notin \qrGroup$ without interactions among the auditors (by locally checking the Jacobi symbol), and ensure that within $\bigO(\secParam)$ tries, the protocol identifies a random generator with overwhelming probability. While the key generation is the most expensive step of our protocol, \emph{this setup is only performed once when the auditors initialize the system}.

\myparagraph{Signing and updating signatures}
%
Signing a set of items $\blocklist$ is significantly cheaper.  We leverage the protocol of \citet[Fig. 3]{catalano2000computing} to compute an inverse modulo a shared order, $\prod\limits_{\genericBlocklistElmt \in \blocklist}\primeRO(\genericBlocklistElmt)^{-1} \bmod \sgPrime[1] \sgPrime[2]$. Knowing a share $\langle \genericSetElmt \rangle_{\auditorIdx}$, the auditors can compute $\sign[\privKey](\blocklist) = \qrGen^{\left(\sum\limits_{\auditorIdx \in [1, \numAuditors]}\langle \genericSetElmt \rangle_{\auditorIdx}\right)}\bmod \rsaMod = \prod\limits_{\auditorIdx = 1}^{\numAuditors} \qrGen^{\langle \genericSetElmt_{\setIdx} \rangle_{\auditorIdx}} \bmod \rsaMod$. Crucially, this product is computed \emph{in the clear} after each auditor broadcasts  $\qrGen^{\langle \genericSetElmt \rangle_{\auditorIdx}} \bmod \rsaMod$. 

%The costs scale polynomially in the size of $\blocklist$,  concretely, it takes 166 seconds to sign a blocklist with 1 million items (see \secref{sec:eval}).

The $\distDel$ can be implemented for ``free'', i.e., without any interaction between the auditors because computing $\genericSig[\blocklist \setminus \genericSetVar] = \genericSig[\blocklist] \sigMinus[\pubKey] \genericSetVar$ does not require any inversions.

\subsection{Content Moderation from Randomizable Set-Homomorphic Signatures}
%
As it might be evident, the randomizable set-homomorphic signature scheme satisfies almost all the 
requirements outlined for a cryptographic token in \secref{sec:model}. We will formally describe such 
a scheme in \secref{sec:protocol}, but will will generally discuss its salient features in context of our 
RSA-based construction. First, note that the blocklist token, i.e., the signature are only two elements 
in $\qrGroup$. Second, the token can be easily updated when new items are added to the blocklist. 
That is, given a blocklist $\blocklist$ and the signature, $\genericSig[\blocklist]$, adding items in a set 
$\blocklist'$ to the blocklist entails computing the token $\genericSig[\blocklist] 
\sigHomomorphism[\pubKey] \genericSig[\blocklist'] = \genericSig[\blocklist \cup \blocklist']$ where 
$\genericSig[\blocklist']$ is the signature on $\blocklist'$. The cost of this operation scales linearly in 
the size of $\blocklist'$, the number of new items to be added, \textit{independent of the size of 
$\blocklist$}.  The remaining two requirements are confidentiality of the blocklist 
against the client, and the accommodating multiple signers. We will describe below how the 
RSA-based scheme can satisfy these requirements. 


\myparagraph{Extending to group of auditors}
%
If there is a group of mutually-untrusting auditors $\auditor[1], \dots, \auditor[\numAuditors]$ that 
will sign the blocklist then the private key 
needs to be shared between the auditors. With the RSA-based signature, we can use a
distributed key generation protocol~\cite{}. More specifically, the auditors generate a shared 
modulus $\rsaMod$, and have 
additive shares of the order of the quadratic residue group $\orderFn(\qrGroup)$. Then, the auditors 
agree on all the items in the blocklist, and for each $\genericBlocklistElmt \in \blocklist$, the auditor 
compute additive shares of $\primeRO(\genericBlocklistElmt)^{-1} \bmod \orderFn(\qrGroup)$. 
Given 
$\primeRO(\genericBlocklistElmt)^{-1}_{\auditorIdx}$ as its share, auditor $\auditor[\auditorIdx]$ 
computes 
$\qrGen^{\primeRO(\genericBlocklistElmt)^{-1}_{\auditorIdx}} \bmod \rsaMod$ and the auditors put 
their shares 
together to compute $\qrGen^{\primeRO(\genericBlocklistElmt)^{-1}} = \prod\limits_{\auditorIdx \in 
	[1, 
	\numAuditors]} \qrGen^{\primeRO(\genericBlocklistElmt)^{-1}_{\auditorIdx}} \bmod \rsaMod$. Note 
that $\qrGen^{\primeRO(\genericBlocklistElmt)^{-1}}$ is the signature on $\genericBlocklistElmt$.

The auditors incrementally compute the signed blocklist as follows. Given the signature on two 
blocklist elements $\genericSig[\genericBlocklistElmt] = 
\qrGen^{\primeRO(\genericBlocklistElmt)^{-1}} \bmod \rsaMod$ and 
$\genericSig[\genericBlocklistElmt'] = \qrGen^{\primeRO(\genericBlocklistElmt')^{-1}} \bmod 
\rsaMod$, a combined signature can be computed with $\genericSig[\genericBlocklistElmt] 
\sigHomomorphism[\pubKey] \genericSig[\genericBlocklistElmt']$, which as defined before in 
\eqnref{eq:sig_homomorphism_union} is a signature on $\genericBlocklistElmt \cup 
\genericBlocklistElmt'$.  


\myparagraph{Blocklist confidentiality}
%
A straightforward solution for confidentiality is to \textit{blind} the signature by computing 
$\genericSig[\blocklist] =  \qrGen^{\left(\frac{1}{\blindingFactor \blocklistAccSet}\right)} \bmod 
\rsaMod$ where $\blindingFactor \getsr \primes$ is a random prime number. With overwhelming 
probability, $\blindingFactor$ is co-prime to $\sgPrime[1],  
\sgPrime[2]$. Now, to mount a dictionary attack, the client needs to guess $\blindingFactor$
which it does with probability at most $\frac{\log \rsaMod}{\rsaMod}$. For verification, the server is 
provided $\blindingFactor$ in the clear.
