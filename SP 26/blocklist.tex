\myparagraph{Content moderation from authenticated blocklists}
%
A secure ACB construction immediately provides a secure realization of our ideal functionality. We outline below

\enumListStart

\subsection{Bloom Filter Based Blocklist}
%
A Bloom filter provides a succinct representation of a set with storage costs that are sublinear in the set size. A Bloom filter is ideally suited for a large blocklist since it is stored on the client. However, a Bloom filter does not provide confidentiality of the blocklisted items, as required by our functionality\footnote{We note that a Bloom filter without confidentiality was considered in prior work~\cite{kulshrestha2021identifying} but assumed to be unsatisfactory.}. 
We introduce confidentiality by constructing the Bloom filter with obfuscated items, while still supporting efficient searching. This is supported using a new cryptographic primitive called an authenticated OPRF. 




\subsubsection{Authenticated OPRF}
%
We define a functionality called \emph{authenticated OPRF}, that extends the definition of an oblivious pseudorandom function (OPRF) to a functionality \authOPRFFunc (\figref{fig:authOPRF_func}), where a group of $\numAuditors$ auditors with semi-honest majority jointly sign a list of inputs to a pseudorandom function. \authOPRFFunc receives the key for the PRF from the client acting as the sender. The receiver of the PRF evaluations is the server, who does not provide inputs.

\myparagraph{Realizing \authOPRFFunc}
%
We realize \authOPRFFunc by leveraging the the 2HashDH PRF construction~\cite{jarecki2014round} in a group \genGrp where the OM gap-DH problem is hard. The construction is $\prf{\prfKeyClient}(\genericFnInput) = \oprfRO[2](\genericFnInput, \oprfRO[1](\genericFnInput)^{\prfKeyClient})$, where $\oprfRO[1]: \universe \rightarrow \genGrp$ and $\oprfRO[2]: \bin^{\setSize{\universe} + \setSize{\genGrp}} \rightarrow \bin^{\secParam}$ are random oracles. The protocol is in \figref{protocol:authOPRF} in the Appendix. In brief,

\enumListStart
\item The auditors ``blind'' the blocklist elements and publish an ordered set $\blindedBlocklist \assign \{\langle \setIdx, \oprfRO[1](\genericBlocklistElmt[\setIdx]\rangle)^{\oprfBlindingFactor[\setIdx]}\}_{\genericBlocklistElmt[\setIdx] \in \blocklist}$. The server is provided the blinding factors and the (ordered) blocklist elements $\langle 1, \genericBlocklistElmt_1, \oprfBlindingFactor[1]\rangle, \dots, \langle \blocklistSize, \genericBlocklistElmt_\blocklistSize, \oprfBlindingFactor[\blocklistSize] \rangle$ . 

\item The auditors sign (a hash of) \blindedBlocklist under their private keys. The client obtains \blindedBlocklist and the signatures. 

\item After verifying the signatures, the client exponentiates the blinded elements in \blindedBlocklist with its PRF key \prfKeyClient and produces another ordered set $\{ \langle \setIdx, \left(\oprfRO[1](\genericBlocklistElmt[\setIdx])^{\oprfBlindingFactor[\setIdx]}\right)^{\prfKeyClient}\rangle\}_{\genericBlocklistElmt[\setIdx] \in \blocklist}$ which is sent to the server. 

\item The server removes the blinding by computing $\left(\left(\oprfRO[1](\genericBlocklistElmt[\setIdx])^{\oprfBlindingFactor[\setIdx]}\right)^{\prfKeyClient}\right)^{\oprfBlindingFactor[\setIdx]^{-1}}$ for each $\{ \langle \setIdx, \left(\oprfRO[1](\genericBlocklistElmt[\setIdx])^{\oprfBlindingFactor[\setIdx]}\right)^{\prfKeyClient}\rangle\}_{\genericBlocklistElmt[\setIdx] \in \blocklist}$. Thus, the server obtains
$\{ \langle \genericBlocklistElmt_\setIdx, \oprfRO[2](\genericBlocklistElmt[\setIdx], \oprfRO[1](\genericBlocklistElmt[\setIdx])^{\prfKeyClient} \rangle \}_{\genericBlocklistElmt[\setIdx] \in \blocklist} = \{ \langle \genericBlocklistElmt_\setIdx, \prf{\prfKeyClient}(\genericBlocklistElmt[\setIdx]) \rangle \}_{\genericBlocklistElmt[\setIdx] \in \blocklist}$. 
\enumListEnd




\subsubsection{Blocklist construction}
%
We create a Bloom-filter-based ACB \bfBlklistToken using \authOPRFFunc as shown in \algoref{acb:bf} and depicted in \figref{fig:bf_acb}.  At a high level, the auditors invoke \authOPRFFunc with \publishList to publish a blinded blocklist \blindedBlocklist, while the client and server invoke \evaluate with \blindedBlocklist as input (line 6). Using the output $\{ \langle \genericBlocklistElmt_\setIdx, \prf{\prfKeyClient}(\genericBlocklistElmt[\setIdx]) \rangle \}_{\genericBlocklistElmt[\setIdx] \in \blocklist}$ from \authOPRFFunc, the server creates $\numBins \assign \lceil \blocklistSize/\log \blocklistSize \rceil$ Bloom filters, $\bloomFilter[1], \dots, \bloomFilter[\numBins]$ , where $\blocklistSize = \setSize{\blocklist}$. The items $\genericBlocklistElmt$ and $\prf{\prfKeyClient}(\genericBlocklistElmt)$ are added to $\bloomFilter[\bloomFilterIdx]$ where $\bloomFilterIdx = \prf{\prfKeyClient}(\genericBlocklistElmt) \bmod \numBins$ (lines 8 - 15).  Instead of directly storing 
$\prf{\prfKeyClient}(\genericBlocklistElmt)$ in the Bloom filter, the server stores $\prp{\prpKeyServer}(\prf{\prfKeyClient})(\genericBlocklistElmt)$, where \prp is a pseudorandom permutation and \prpKeyServer is a key sampled by the server; the reason will be clarified in \secref{sec:content_mod}. Finally, the Bloom filters are encrypted by the server under a symmetric key and stored by the client as the ACB. 

During a search with input $\genericCInputElmt$, the client sends to the server $\genericCInputElmtPRFEval \assign \prf{\prfKeyClient}(\genericCInputElmt)$ and the encrypted Bloom filter $\bloomFilter[\bloomFilterIdx]$ where $\bloomFilterIdx = \genericCInputElmtPRFEval \bmod \numBins$. The server checks if $\prp{\prpKeyServer}(\genericCInputElmtPRFEval) \in \bloomFilter[\bloomFilterIdx]$, and if so, the search returns a positive (lines 21 - 22). To prevent the server from correlating searches, the client maintains a list of previously queried items, \prevHashList. If $\genericCInputElmt \in \prevHashList$, then the client sets $\genericCInputElmtPRFEval \sample \bin^{\secParam}$.

\myparagraph{Costs}
%
The costs of creating and searching are as follows.

\itemListStart
\item \createBlocklist has $\bigO(\blocklistSize)$ cost for the auditors, client and server. This is a one-time cost.
\item \searchBlocklist has $\bigO(1)$ cost for the client and the server, and $\bigO(\log \blocklistSize)$ communication cost for a single Bloom filter. 
\item The total storage cost for the client is $\sum\limits_{\bloomFilterIdx \in [1, \numBins]} |\bloomFilter[\bloomFilterIdx]|$.
\itemListEnd 

\myparagraph{Security}
%
The security of the ACB is due to the following.

\itemListStart

\item During a search, the server learns $\prf{\prfKeyClient}(\genericCInputElmt)$, which is indistinguishable from random to the server if $\genericCInputElmt \notin \blocklist$. \subsection{Homomorphic Signature Based Blocklist}
%
There are two drawbacks of the Bloom filter based ACB: i) the storage overhead for \blklistToken is $\bigO(\blocklistSize)$, and ii) the search can produce false positives. In this section, we will design an ACB that mitigates both of these problems, but with higher computational costs for the server during search. The construction leverages a new cryptographic primitive called a \emph{randomizable set homomorphic signature}. Intuitively, the ACB is a signature on the accumulated set of blocklisted items~\cite{johnson2002homomorphic} under the auditors' private keys. The signature scheme has homomorphic properties that allows us to compute signatures of subsets of a signed set, as well as the union of two signed sets without the private key. We leverage these properties to directly compute on the signature, and implement an oblivious match. Importantly, the signature (and the ACB) is constant-sized making it storage efficient.


% Moreover, the ACB is efficiently updateable with costs scaling linearly in the number of additions/deletions, and \emph{independent of size of the modified blocklist}.

\myparagraph{Security}
%
A randomizable set homomorphic signature scheme must satisfy three security properties

\itemListStart
\item \textbf{Unforgeability:} The signature scheme must satisfy unforgeability under chosen message attacks (UF-CMA), defined by the experiment $\Experiment{\ufcmaLabel}{\sigScheme}(\ufcmaAdversary)$ in \figref{fig:experiments}.  
\item \textbf{Indistinguishable signatures under randomization:}  For all 
$\genericCInputElmt \notin \blocklist$, we want the output of $\randomize[\pubKey](\genericSig[\blocklist] \sigMinus[\pubKey] 
\setDefn{\genericCInputElmt})$ to hide 
$\genericCInputElmt$. We formalize this property as \emph{indistinguishable signatures under randomizations (IND-SR)}, using a CPA-style definition such that for $\genericCInputElmt{0}, \genericCInputElmt{1} \notin \blocklist$,
the outputs of $\randomize[\pubKey](\genericSig[\blocklist]
\sigMinus[\pubKey] \{\genericCInputElmt{0}\})$ and
$\randomize[\pubKey](\genericSig[\blocklist] \sigMinus[\pubKey]
\{\genericCInputElmt{1}\})$ are indistinguishable. This is defined by the experiment $\Experiment{\indsrLabel[\indBit]}{\sigScheme}$ in \figref{fig:experiments}. 

\item \textbf{Confidentiality under blinding:} A blinded signature $\blind(\genericSig[\blocklist], \blindingFactor)$ must not reveal any item in \blocklist, i.e., for any PPT. adversary 


\begin{gather*}
\cprob{bigg}{\begin{array}{c} \genericBlocklistElmt \assign \adversary(\genericSigPair[1])  \\ \land ~\genericBlocklistElmt \in \blocklist\end{array} }{\begin{array}{c} \langle \pubKey, \privKey \rangle \assign \keyGen(1^{\secParam}) \\ \blindingFactor \sample \genericRandSpace \\ \genericSigPair[1] \assign \blind[\privKey](\genericSig[\blocklist], \blindingFactor)\end{array} } \\ \le 1/|\universe| + \neglProb 
\end{gather*}

\itemListEnd

\myparagraph{Correctness}
%
The correctness of $\keyGen$, $\signingAlg$ and $\verify$ is obvious. We describe how the signature scheme satisfies the homomorphic properties. All calculation are performed modulo $\rsaMod$; we omit this factor from the description for notational simplicity. 

Let  $\genericSetVar[1]$, $\genericSetVar[2]$ be two sets and $\accSet[1] \gets 
\prod\limits_{\genericCInputElmt \in \genericSetVar[1]} \primeRO(\genericCInputElmt)$, and $\accSet[2] \gets 
\prod\limits_{\genericCInputElmt' \in \genericSetVar[2]}  \primeRO(\genericCInputElmt')$. Also, let the signatures on $\genericSetVar[1]$ and $\genericSetVar[2]$ be $\genericSig[\genericSetVar[1]] \assign \qrGen^{1/\accSet[1]}$ and $\genericSig[\genericSetVar[2]] \assign \qrGen^{1/\accSet[2]}$ respectively. %Now, note that


\itemListStart
\item \itemHeading{Union of sets:} We define the homomorphic operator $\sigHomomorphism$ 


\begin{align*}
& \genericSig[\genericSetVar[1]] \sigHomomorphism[\pubKey] \genericSig[\genericSetVar[2]] = \genericSig[\genericSetVar[1] \cup \genericSetVar[2]] = 
\genericSig[\genericSetVar[1]]^{\euclideanConst[2]}  \genericSig[\genericSetVar[2]]^{\euclideanConst[1]} = \qrGen^{\euclideanConst[2]/\accSet[1]} \qrGen^{\euclideanConst[1]/\accSet[2]} \\
&  = \qrGen^{\frac{\euclideanConst[2] \accSet[2] + \euclideanConst[1] \accSet[1]}{\accSet[1] \accSet[2]}} =\qrGen^{\frac{\gcd(\accSet[1], \accSet[2])}{\accSet[1] \accSet[2]}} = \qrGen^{1/\lcm(\accSet[1], \accSet[2])} \label{eq}
\end{align*}


Here, $\euclideanConst[1], \euclideanConst[2]$ are integers such that $\euclideanConst[1]\accSet[1] + \euclideanConst[2] \accSet[2] = \gcd(\accSet[1], \accSet[2])$, and they can be found in polynomial time using the extended Euclid algorithm. Due to Bezout's identity $\euclideanConst[1], \euclideanConst[2]$ always exist. 






\item \itemHeading{Difference of two sets:} We define the set difference operator $\sigMinus$ 

\[
\genericSig[\genericSetVar[1]] \sigMinus[\pubKey] \genericSetVar[2] = \genericSig[\genericSetVar[1] \setminus \genericSetVar[2]] = \genericSig[\genericSetVar[1]]^{\accSet[2]} = \qrGen^{\accSet[2]/\accSet[1]}
\]

, where $\genericSetVar[2] \subseteq \genericSetVar[1]$, 


\item \itemHeading{Homomorphic property over randomization:} The set difference operator $\sigMinus$ is homomorphic over the randomization function.    


\begin{align*} 
& \randomizeFn[\pubKey](\genericSig[\genericSetVar[1]], \genericRand) 
\sigMinus[\pubKey] \genericSetVar[2] = \qrGen^{\genericRand \fieldMult \accSet[2]/\accSet[1]}  = \\
&\randomizeFn[\pubKey]\left(\qrGen^{\accSet[2]/\accSet[1]}, \genericRand\right) = \randomizeFn[\pubKey](\genericSig[\genericSetVar[1]] \sigMinus[\pubKey] \genericSetVar[2], \genericRand)
\end{align*}

\itemListEnd

strong QR-RSA assumption in the random oracle model. Confidentiality under blinding is ensured by the fact that for $\blindingFactor \sample [1, \rsaMod^2]$, $\randomizeFn(\genericSigPair[1], \blindingFactor) = \genericSigPair[1]^{\blindingFactor} \bmod \rsaMod$ is uniformly distributed in \qrGroup~\cite{micciancio2005rsa}. \myparagraph{Distributed Key Generation \& Signing}

To leverage the signature scheme with multiple signers, as required to publish an ACB, we need to extend it to a distributed setting such that the auditors only have a share of the private key. Additionally, the distributed key generation must be robust against a malicious minority of signers in our threat model. Otherwise, the server learns nothing due to the IND-SR property of the signature scheme.

\subsection{Encrypted Cuckoo Hash Based Blocklist}
%
The blocklist constructions we have considered so far only support the exact search (\searchBlocklist) mechanism. In this section, we will design an ACB that supports threshold search (\thresholdSearchBlocklist) mechanism. Intuitively, the blocklisted items are hashed into a cuckoo hash table \cuckooTable of size $\blocklistSize \numProjections(1+\cuckooTableParam)$ with hash functions $\cuckooHashFn{1}, \cuckooHashFn{2}$, where $\cuckooTableParam$ is a parameter that controls false negative rate of the cuckoo hash table~\cite{bhowmick2021apple}. \cuckooTable is stored on the client and the values in each location is individually encrypted under an additively homomorphic encryption scheme. \emph{The hash keys are not stored alongside}: the client sees $2 \blocklistSize \numProjections(1+\cuckooTableParam)$ encrypted values, and knows the relation between ciphertexts and their corresponding locations in the table. The empty locations in the table are padded with encryptions of dummy values. 


During a threshold search the client performs lookups in \cuckooTable, and then compares its inputs with the encrypted values in the corresponding locations in the hash table, by homomorphically subtracting one from the other, and blinding the result with a random multiplicative factor. If the projections match, then after decryption, the server should observe an encryption of \groupIdentity. 

To support threshold searches, the client secret-shares a known value in the encrypted results provided to the server. After decryption, if the server obtains enough shares, then it can reconstruct the value and confirm. \emph{The main challenge is that the server should know which shares have been correctly decrypted when there are at least a threshold number of them, otherwise  the server should not learn any information}. We address this challenge by using a \emph{detectable} secret sharing scheme. We provide the necessary background below.

\subsubsection{Detectable Secret Sharing}
%
Similar to standard $(\secretShareLength, \secretShareThreshold)$-secret-sharing, a detectable secret sharing (DSS) scheme has two algorithms $\detectableSecretSharingScheme = \langle \DSShare, \DSCombine\rangle$. 

\itemListStart
\item $\{ \secretShare{1},\dots, \secretShare{\secretShareLength} \} \gets \DSShare(\genericSymmKey)$ takes as input a secret $\genericSymmKey$ and produces a set of secret-shares $\{ \secretShare{1},\dots, \secretShare{\secretShareLength} \}$
\item $\genericSymmKey/\bot \gets \dssProtocol.\DSCombine(\{\genericSetElmt[1], \dots, \genericSetElmt[\secretShareLength] \})$ Given a set of $\secretShareLength$ elements, if at least \secretShareThreshold of them are valid secret shares of the same secret, and the rest are random elements, output the secret $\genericSymmKey$. Otherwise, output $\bot$. 
\itemListEnd

For security, a DSS scheme ensures that when there are less that \secretShareThreshold valid shares of \genericSymmKey in $\{\genericSetElmt[1], \dots, \genericSetElmt[\secretShareLength] \}$ then nothing is revealed to an adversary \emph{including which of the shares are correct} in the set $\{\genericSetElmt[1], \dots, \genericSetElmt[\secretShareLength] \}$



\myparagraph{Detectable hash function}
%
We will make use of a primitive introduced by the Apple PSI mechanism~\cite{bhowmick2021apple} called a detectable hash 
function to realize a DSS scheme. Formally, a $(\dhfParamMatchThreshold, \dhfParamRandThreshold)$-detectable 
hash function $\dhf: \dhfKeySpace \times \dhfDomain \rightarrow \dhfRange$ has the following 
two properties.

\enumListStart
\item The function is weakly \textit{\dhfParamMatchThreshold-wise
  independent}, that is the distribution of
  $\langle\dhf[\dhfKey](\genericFnInput[1]), \dots,
  \dhf[\dhfKey](\genericFnInput[\dhfParamMatchThreshold])\rangle$
  given $\dhfKey \getsr \dhfKeySpace$ and $\genericFnInput[1], \dots,
  \genericFnInput[\dhfParamMatchThreshold] \getsr \dhfDomain$ is
  identical to the uniform distribution over
  $(\dhfDomain)^{\dhfParamMatchThreshold}$.
	
\item The function is \textit{\dhfParamRandThreshold-detectable}. 
  Given a vector $\genericVec = \tupleDefn{\genericVec[1], \dots, 
    \genericVec[\dhfLength]} \in (\dhfDomain)^{\dhfLength}$, where for $\matchedIndexSet 
  \subseteq \nats[\dhfLength]$, $\genericVec[\projIdx] \gets 
  \dhf[\dhfKey](\genericFnInput[\projIdx])$ when $\projIdx \in \matchedIndexSet$, 
  and $\genericVec[\projIdx] \getsr \dhfDomain$ when $\projIdx \notin \matchedIndexSet$, 
  there is an algorithm $\dhfDetect$   
	\itemListStart
  \item $\dhf[\dhfKey].\dhfDetect(\genericVec) =
    \matchedIndexSet$ with non-negligible probability,
    when $\setSize{\matchedIndexSet} \ge
    \max(\dhfParamMatchThreshold+1, \genericVecLen -
    \dhfParamRandThreshold)$.
  \item $\dhf[\dhfKey].\dhfDetect(\genericVec) =
    \dhfFail$ when $\setSize{\matchedIndexSet} \le
    \dhfParamMatchThreshold$.
\itemListEnd
\enumListEnd
  

  Intuitively, given a vector $\genericVec \in
  (\dhfDomain)^{\dhfLength}$, where at least
  $\max(\dhfParamMatchThreshold+1, \dhfLength - \dhfParamRandThreshold)$ 
  components are outputs of $\dhf$,
  the detection algorithm identifies these components. However,
  if less than $\dhfParamMatchThreshold+1$ entries are generated
  by $\dhf$, then the detection algorithm does not reveal any
  information about $\matchedIndexSet$ due to the \dhfParamMatchThreshold-wise independence. More details about the DHF scheme is in \cite{bhowmick2021apple} is in \appref{}.

\subsubsection{Blocklist construction}
%
The ACB is the encrypted cuckoo hash table. The full construction is described in \algoref{acb:cht} and depicted in \figref{fig:cht_acb}. Each location in the table is an encrypted item in the blocklist, as decided by the cuckoo hash functions (lines 3 - 5).

During a search $\thresholdSearchBlocklist(\genericCInputSet, \chtBlklistToken, \threshold)$, the client uses a detectable secret sharing scheme such that the server is able to reconstruct a secret only when $\setSize{\genericCInputSet \cap \blocklist} \ge \threshold$. This is enabled by encrypting the detectable secret shares of the secret with independently sampled keys for a PRG (\eqnsref{cht:homomorphic_add_1}{cht:homomorphic_add_2}), and encrypting the PRG keys using an additively homomorphic encryption such that the server obtains a key if the client's input matches a corresponding (encrypted) item in the table (\eqnsref{cht:prg_1}{cht:prg_2}). If the server obtains at least $\threshold$ PRG keys, then it can decrypt $\threshold$ DSS components, allowing it to reconstruct the secret (lines 15 - 17). 


\myparagraph{Cost \& Security}
%
The cost of creating the blocklist with \blocklistSize items is $\bigO(\blocklistSize)$. The cost of a threshold search is $\bigO(\secretShareLength)$ for the client and $\bigO(\secretShareLength \times \secretShareThreshold )$ cost for the server due to the \dhfDetect algorithm. The security of the scheme is due to the IND-CPA security of the additively homomorphic encryption scheme. The secret sharing scheme, and the DHF guarantees are both information-theoretically secure.

\newcommand{\prgSeedAlt}[1]{\ensuremath{\kappa^{'}_{#1}}\xspace}

\begin{algorithm}
\caption{\chtACB: Encrypted Hash Table Based Blocklist}
\label{acb:cht}
\footnotesize
\begin{algorithmic}[1]

\Require A signature scheme \sigSchemeDefn; a collision-resistant hash function $\bin^{\ast} \rightarrow \bin^{\secParam}$; an additively homomorphic encryption scheme \ahencryptionSchemeDefn; a $(2\genericVecLen, \threshold)$ detectable secret sharing scheme $\detectableSecretSharingSchemeDefn$; a pseudorandom generator $\prg: \prgKeySpace \rightarrow \bin^{\secParam}$

\Statex \Procedure{\publishList}{\blocklist}

\State Auditor $\auditor[\auditorIdx]$ samples a key pair $\langle \auditorPubKeyGenericScheme[\auditorIdx], \auditorPrivKeyGenericScheme[\auditorIdx] \rangle$ for \sigScheme


\State The auditors initialize a cuckoo hash table \cuckooTable with $(1 + \cuckooTableParam) \cdot \blocklistSize$ slots, and samples two hash function $\cuckooHashFn{1}, \cuckooHashFn{2}: \bin^{\secParam} \rightarrow \nats[(1 + \cuckooTableParam) \cdot \blocklistSize]$. 

\State The auditors samples a key pair $\encryptPubKey, \encryptPrivKey$ for \ahencryptionScheme


\State For $\genericBlocklistElmt \in \blocklist$, the auditors place $\ahencrypt[\encryptPubKey](\genericBlocklistElmt)$ at slot $\cuckooTable[\cuckooHashFn{1}(\projFn[\projIdx](\genericBlocklistElmt))]$ or $\cuckooTable[\cuckooHashFn{2}(\projFn[\projIdx](\genericBlocklistElmt))]$ using cuckoo hashing. At each empty slot in the table, store $\ahencrypt[\encryptPubKey](\groupIdentity)$.

\State The auditors sign the ACB $\chtBlklistToken \assign \cuckooTable$ under their own private keys and publish $\{\signGenSig[\auditorPrivKeyGenericScheme[\auditorIdx]](\hashFn(\blklistToken))\}_{\auditorIdx \in [1, \numAuditors]}$. The auditors provide \blocklist and \encryptPrivKey to \server. 


\EndProcedure


\Statex\Procedure{\thresholdSearchBlocklist}{$\genericCInputSet \assign \{ \genericCInputElmt{\setIdx}\}_{\setIdx \in [1, \genericVecLen]}, \chtBlklistToken, \threshold$}


\LComment{Client secret-shares a secret value and encrypts the shares}

\State \client invokes $\ssSet \assign \DSShare(\groupIdentity)$
\State \client samples 2\genericVecLen PRG keys $\prgSeed{1}, \dots, \prgSeed{2\genericVecLen} \sample \prgKeySpace$

\For{$\genericCInputElmt{\projIdx} \in \genericCInputElmt{1}, \dots, \genericCInputElmt{\genericVecLen}$}

\State \client retrieves $\ciphertext[1]$ from $\cuckooTable[\cuckooHashFn{1}(\genericCInputElmt{\projIdx})]$ and 
$\ciphertext[2]$ from $\cuckooTable[\cuckooHashFn{2}(\genericCInputElmt{\projIdx})]$ and computes for $\ssSet = \{ s_1^{\ast}, \dots, s_{2\genericVecLen}^{\ast} \}$ and $\projIdx \in [1, \genericVecLen]$

\begin{align}
& \secretShareEncryption{1, \projIdx} \assign \genericRand[1, \projIdx] \homomorphicScalerMult{\encryptPubKey} (\ciphertext[1] \homomorphicAdd{\encryptPubKey}  ( - \genericCInputElmt{\projIdx})) \homomorphicAdd{\encryptPubKey} \pkencrypt[\encryptPubKey](\prgSeed{\projIdx})
\label{cht:homomorphic_add_1} \\ 
& \secretShareEncryption{2, \projIdx} \assign \genericRand[2, \projIdx] \homomorphicScalerMult{\encryptPubKey} (\ciphertext[2] \homomorphicAdd{\encryptPubKey}  ( - \genericCInputElmt{\projIdx})) \homomorphicAdd{\encryptPubKey} \pkencrypt[\encryptPubKey](\prgSeed{\genericVecLen + \projIdx})
\label{cht:homomorphic_add_2} \\ 
& \secretShareEncryption{1, \projIdx}^{\ast} \assign s_{\projIdx}^{\ast} \oplus \prg(\prgSeed{\projIdx}) \label{cht:prg_1}\\
& \secretShareEncryption{2, \projIdx}^{\ast} \assign s_{\genericVecLen + \projIdx}^{\ast} \oplus \prg(\prgSeed{\genericVecLen + \projIdx})\label{cht:prg_2} 
\end{align}

where $\genericRand[1, \projIdx], \genericRand[2, \projIdx] \sample [1, \groupOrder - 1]$. The encrypted calculations in \eqnsref{cht:homomorphic_add_1}{cht:homomorphic_add_2} are performed using the additively homomorphic operators defined in \secref{sec:def}. 


\EndFor

\State \client sends to \server $\{\secretShareEncryption{\tblIdx,\projIdx}\}_{\tblIdx \in [2], \projIdx \in [1, \genericVecLen]}$ and $\{\secretShareEncryption{\tblIdx,\projIdx}^{\ast}\}_{\tblIdx \in [2], \projIdx \in [1, \genericVecLen]}$


\Statex \LComment{Upon receiving the above inputs from \client}


\State \server decrypts $\secretShareEncryption{i, \projIdx}$ for all $i \in [1,2]$ and $\projIdx \in [1,2\genericVecLen]$ using \encryptPrivKey, and obtains $\{ \prgSeedAlt{1}, \dots, \prgSeedAlt{2\genericVecLen} \}$

\State \server computes 

\begin{multline*}
\ssSet' \assign \{ s': \secretShareEncryption{1, \projIdx}^{\ast} \oplus \prgSeed{\projIdx}, \projIdx \in [1, \genericVecLen]\} \\
\bigcup \{ s': \secretShareEncryption{2, \projIdx}^{\ast} \oplus \prgSeed{\genericVecLen + \projIdx}, \projIdx \in [\genericVecLen + 1, 2\genericVecLen] \}
\end{multline*}


\State If $\DSCombine(\ssSet') = \bot$, then \server outputs $\fals$. Otherwise it outputs $\tru$. 

%$\langle \genericFileID, \detected = \fals, \emptyset \rangle$ and sends to \client $\langle \genericFileID, \detected=\fals\rangle$. This concludes the fast path check. 


\EndProcedure




\end{algorithmic}
\end{algorithm}
