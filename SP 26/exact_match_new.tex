\section{Exact Match Without Fast Path}
%

In this section, we will establish the foundations of our content moderation scheme for exact matching. In the next sections, we will include new components to support a \emph{fast path} for content verification, and extend the scheme to support fuzzy matching. The core building block of our scheme is a new cryptographic primitive called a \emph{randomizable set-homomorphic signature}. 

\subsection{Building Block: Randomizable Set-Homomorphic Signatures}
%
Recall that we need to design the blocklist authentication token \blklistToken{\blocklist} and content verification token \contentToken{\clientData} such that they satisfy the security requirements discussed in \secref{sec:model}. Since blocklists can be large~\cite{content_mod_e2ee},  we will ensure that: i) $\blklistToken{\blocklist}$ is sublinear in the blocklist size as the client must store/download the signature, and ii) the client can compute \contentToken{\clientData} with sublinear costs in the blocklist size.

%, and iii) the entire process is non-interactive and requires communication that is 
%sublinear in the blocklist size.  

%Unfortunately, existing designs do not optimize all three metrics (see \tblref{tab:related}). 

We propose a novel cryptographic primitive called a \textit{randomizable set homomorphic signature} to address these goals. Intuitively, we will use a signature by the auditors on the blocklist as \blklistToken{\blocklist} since it is unforgeable. Then  our key idea is to design a scheme that directly allows computation on the signature, in order to compute \contentToken{\clientData}. If the size of the signature scales sublinearly in the blocklist size, and the computation to derive \contentToken{\clientData} scales linearly in the size of the signature (and the client's input), then we achieve a scheme with sublinear compute costs for the client.

\begin{remark}
In our discussions we will denote a signature on the set $\genericSetVar \subseteq\universe$ as $\genericSig[\genericSetVar]$ for notational simplicity. For any $\genericCInputElmt \in \universe$, the computation $\genericSig[\blocklist] \sigMinus[\pubKey] \{\genericCInputElmt\}$ returns a valid signature on the set $\genericSetDiff[\blocklist][\genericCInputElmt]$ when $\genericCInputElmt \in \blocklist$ due to \eqnref{eq:sig_homomorphism_minus}. We denote this signature with $\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]]$. 
 However, when $\genericCInputElmt \notin \blocklist$, $\genericSig[\blocklist] \sigMinus[\pubKey] \{\genericCInputElmt\}$ is an element in the output space $\sigOutputSpace$ that is not necessarily equal to $\genericSig[\blocklist]$. As will be evident later, 
$\genericSig[\blocklist] \sigMinus[\pubKey] \genericCInputElmt$ is a signature on an unpredictable element in \sigOutputSpace, and with abuse of notation we also denote this element as $\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]]$. 
\end{remark}

\myparagraph{Towards a content moderation scheme}
%
We will use a randomizable set-homomorphic signature scheme to build our content moderation scheme (see \figref{fig:sig_use}). Thus, we present the security definitions of the signature scheme in the context of our application.  

\smallskip
\noindent\textit{\underline{Computing the blocklist verification token:}} We will 
use a (blinded) signature $\genericSig[\blocklist]$ on the blocklist as the blocklist authentication token \blklistToken{\blocklist}. 
This token must be unforgeable, so the signature scheme \ourSigScheme must satisfy unforgeability under chosen message attacks (UF-CMA), defined by the experiment $\Experiment{\ufcmaLabel}{\sigScheme}(\ufcmaAdversary)$ in \figref{fig:experiments}.

\smallskip
\noindent\textit{\underline{Computing the content verification token:}}
%
We will use the homomorphic operator to derive the content verification token. When uploading content, \clientData, with content hash \genericCInputElmt, the client will invoke $\randomize[\pubKey](\genericSig[\blocklist] \sigMinus[\pubKey] 
\setDefn{\genericCInputElmt}) = \langle \randomizeFn[\pubKey](\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]], \genericRand)$,  $\ro(\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand)) \rangle$. 

\itemListStart
\item \textbf{Content verification token:} The client sets the content verification token $\contentToken{\clientData} \assign  \randomizeFn[\pubKey](\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]], \genericRand)$. Observe that if the blocklist \blocklist is known (as is the case for the server), and  $\genericCInputElmt \in \blocklist$, 
we can calculate
$\randomizeFn[\pubKey](\genericSig[\genericSetDiff[\blocklist][\genericCInputElmt]] 
\sigMinus[\pubKey] \{\genericSetDiff[\blocklist][\genericCInputElmt]\}, \genericRand) = 
\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand)$ using 
\eqref{eq:homomorphic_randomizeFn} by iterating over the items in \blocklist. 

%Here, $\genericSig[\emptyset]$ 
%is the signature on the null set, and \emph{is fixed for a given public key}. Also, since $\randomizeFn[\pubKey]$ is injective for a fixed $\genericRand$, $\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand)$ is unique. 

\item \textbf{Encryption key:} The client encrypts the key under which $\clientData$ is encrypted as $\genericSymmKey \oplus \ro(\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand))$ assuming that $\ro$ provides a random output of length equal to $\setSize{\encryptKeySpace}$. If the server is able to derive $\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand)$ using the $\sigMinus[\pubKey]$ operator from the $\contentToken{\clientData}$ when $\genericCInputElmt \in \blocklist$ as described above, then it can also derive $\genericSymmKey$ and decrypt the content.

\itemListEnd


For security, we ensure two properties. First, when 
$\genericCInputElmt \notin \blocklist$, we want the output of $\randomize[\pubKey](\genericSig[\blocklist] \sigMinus[\pubKey] 
\setDefn{\genericCInputElmt})$ to hide 
$\genericCInputElmt$ from the server. We formalize this property as \emph{indistinguishable signatures under randomizations (IND-SR)}, using a CPA-style definition such that for $\genericCInputElmt{0}, \genericCInputElmt{1} \notin \blocklist$,
the outputs of $\randomize[\pubKey](\genericSig[\blocklist]
\sigMinus[\pubKey] \{\genericCInputElmt{0}\})$ and
$\randomize[\pubKey](\genericSig[\blocklist] \sigMinus[\pubKey]
\{\genericCInputElmt{1}\})$ are indistinguishable. This is defined by the experiment $\Experiment{\indsrLabel[\indBit]}{\sigScheme}$ in \figref{fig:experiments}. Second, when $\genericCInputElmt \notin \blocklist$, we want $\contentToken{\clientData}$ to hide the encryption key in $\genericSymmKey \oplus \ro(\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand))$. This property is directly implied by the random oracle assumption: since $\ro$ is a random oracle and the output of $\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand)$ is uniformly distributed for a random $\genericRand$, the adversary learns $\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand)$ from $\genericSymmKey \oplus \ro(\randomizeFn[\pubKey](\genericSig[\emptyset], \genericRand))$ only with negligible probability (assuming collisions are negligible).





\subsubsection{RSA-Based Construction}

We describe a randomizable set homomorphic signature scheme in \figref{sig:construction} that is secure under the strong QR-RSA assumption.

	$\genericSig[\genericSetVar]^{\accSet} \equiv \qrGen \bmod

\myparagraph{Correctness}
%
The correctness of $\keyGen$, $\signingAlg$ and $\verify$ is obvious. We describe how the signature scheme satisfies the homomorphic properties. All calculation are performed modulo $\rsaMod$; we omit this factor from the description for notational simplicity. 

Let  $\genericSetVar[1]$, $\genericSetVar[2]$ be two sets and $\accSet[1] \gets 
\prod\limits_{\genericCInputElmt \in \genericSetVar[1]} \primeRO(\genericCInputElmt)$, and $\accSet[2] \gets 
\prod\limits_{\genericCInputElmt' \in \genericSetVar[2]}  \primeRO(\genericCInputElmt')$. Also, let the signatures on $\genericSetVar[1]$ and $\genericSetVar[2]$ be $\genericSig[\genericSetVar[1]] \assign \qrGen^{1/\accSet[1]}$ and $\genericSig[\genericSetVar[2]] \assign \qrGen^{1/\accSet[2]}$ respectively. %Now, note that


\itemListStart
\item \itemHeading{Union of sets:} We define the homomorphic operator $\sigHomomorphism$ below


\begin{align*}
& \genericSig[\genericSetVar[1]] \sigHomomorphism[\pubKey] \genericSig[\genericSetVar[2]] = \genericSig[\genericSetVar[1] \cup \genericSetVar[2]] = 
\genericSig[\genericSetVar[1]]^{\euclideanConst[2]}  \genericSig[\genericSetVar[2]]^{\euclideanConst[1]} = \qrGen^{\euclideanConst[2]/\accSet[1]} \qrGen^{\euclideanConst[1]/\accSet[2]} \\
&  = \qrGen^{\frac{\euclideanConst[2] \accSet[2] + \euclideanConst[1] \accSet[1]}{\accSet[1] \accSet[2]}} =\qrGen^{\frac{\gcd(\accSet[1], \accSet[2])}{\accSet[1] \accSet[2]}} = \qrGen^{1/\lcm(\accSet[1], \accSet[2])} \label{eq}
\end{align*}


Here, $\euclideanConst[1], \euclideanConst[2]$ are integers such that $\euclideanConst[1]\accSet[1] + \euclideanConst[2] \accSet[2] = \gcd(\accSet[1], \accSet[2])$, and they can be found in polynomial time using the extended Euclid algorithm. Due to Bezout's identity $\euclideanConst[1], \euclideanConst[2]$ always exist. 






\item \itemHeading{Difference of two sets:} We define the set difference operator $\sigMinus$ below

\[
\genericSig[\genericSetVar[1]] \sigMinus[\pubKey] \genericSetVar[2] = \genericSig[\genericSetVar[1] \setminus \genericSetVar[2]] = \genericSig[\genericSetVar[1]]^{\accSet[2]} = \qrGen^{\accSet[2]/\accSet[1]}
\]

, where $\genericSetVar[2] \subseteq \genericSetVar[1]$, 


\item \itemHeading{Homomorphic property over randomization:} The set difference operator $\sigMinus$ is homomorphic over the randomization function.    


\begin{align*} 
& \randomizeFn[\pubKey](\genericSig[\genericSetVar[1]], \genericRand) 
\sigMinus[\pubKey] \genericSetVar[2] = \qrGen^{\genericRand \fieldMult \accSet[2]/\accSet[1]}  = \\
&\randomizeFn[\pubKey]\left(\qrGen^{\accSet[2]/\accSet[1]}, \genericRand\right) = \randomizeFn[\pubKey](\genericSig[\genericSetVar[1]] \sigMinus[\pubKey] \genericSetVar[2], \genericRand)
\end{align*}

\itemListEnd


We show in \appref{sec:proof:sig_scheme} that the security of the scheme reduces to the 
strong QR-RSA assumption in the random oracle model.

\begin{remark}
The size of a signature of a set $\blocklist$ under \ourSigScheme is \emph{independent} of the size of $\blocklist$ and is only a single element in $\qrGroup$. Thus, our blocklist authentication token, which is simply the signature $\genericSig[\blocklist]$ on the blocklist, is constant sized. Moreover, the content verification token is derived from the output of $\randomize(\genericSig[\blocklist] \sigMinus[\pubKey] \genericCInputElmt)$ which is a constant time operation and outputs two elements in \qrGroup. As a result, the cost of computing the content verification token is \emph{independent} of the blocklist, unlike some existing designs~\cite{kulshrestha2021identifying}. 
\end{remark}

\subsubsection{Distributed Key Generation \& Signing}
%
In our description so far, we have assumed that there is a single party with the private key. However, to leverage the signature scheme in our protocol, where multiple auditors sign the blocklist, we need to extend it to a distributed setting such that the auditors only have a share of the private key. Additionally, the distributed key generation and sharing must be robust against a malicious minority of signers. Fortunately, our signature scheme can be extended to support distributed key generation and singing using existing techniques. Due to space constraints, we define the ideal functionality \distSigningFunc for distributed key generating and singing, and describe the protocol realizing the functionality in \appref{app:dist_signing}. We provide a high level overview of the solution below. 

\myparagraph{Key generation}
%
Recall that the public key of the signature scheme is the modulus $\rsaMod$ and a generator $\qrGen$, and the private key is the factorization of $\rsaMod$. We use the protocol of \citet{algesheimer2002efficient}\footnote{To the best of our knowledge this is the only protocol that ensures that $\rsaMod$ is generated from safe primes, which is necessary to ensure that $\qrGroup$ is cyclic} to generate a shared modulus $\rsaMod = (2 \sgPrime[1] + 1) (2 \sgPrime[2] + 1)$ such that the auditors only have additive shares of $\sgPrime[1], \sgPrime[2]$.  
 The shares allow the auditors to securely compute $\rsaMod$ and $\orderFn(\qrGroup) = \sgPrime[1] \sgPrime[2]$ using the BGW protocol~\cite{asharov2017full}. \rsaMod is included in the public key while shares of $\orderFn(\qrGroup)$ are distributed to the auditors and kept as the private key. 

The generator selection is also performed using the BGW protocol by checking that for a random value $\qrGen \in [2, \rsaMod -1]$, selected by the auditors, $\qrGen^{\sgPrime[1]} \not \equiv 1 \bmod \rsaMod ~\land ~\qrGen^{\sgPrime[2]} \not \equiv 1 \bmod \rsaMod$. 
It is necessary to perform this check without revealing the outputs of   
$\qrGen^{\sgPrime[1]}, \qrGen^{\sgPrime[2]}$ in the clear, and thus we use the BGW instantiations. We introduce several optimizations to reduce the number of BGW invocations by trivially excluding cases where $\qrGen \notin \qrGroup$ without interactions among the auditors (by locally checking the Jacobi symbol), and ensure that within $\bigO(\secParam)$ tries, the protocol identifies a random generator with overwhelming probability. While the key generation is the most expensive step of our protocol, \emph{this setup is only performed once when the auditors initialize the system}.

\myparagraph{Signing and updating signatures}
%
Signing a set of items $\blocklist$ is significantly cheaper.  We leverage the protocol of \citet[Fig. 3]{catalano2000computing} to compute an inverse modulo a shared order, $\prod\limits_{\genericBlocklistElmt \in \blocklist}\primeRO(\genericBlocklistElmt)^{-1} \bmod \sgPrime[1] \sgPrime[2]$. Knowing a share $\langle \genericSetElmt \rangle_{\auditorIdx}$, the auditors can compute $\sign[\privKey](\blocklist) = \qrGen^{\left(\sum\limits_{\auditorIdx \in [1, \numAuditors]}\langle \genericSetElmt \rangle_{\auditorIdx}\right)}\bmod \rsaMod = \prod\limits_{\auditorIdx = 1}^{\numAuditors} \qrGen^{\langle \genericSetElmt_{\setIdx} \rangle_{\auditorIdx}} \bmod \rsaMod$. Crucially, this product is computed \emph{in the clear} after each auditor broadcasts  $\qrGen^{\langle \genericSetElmt \rangle_{\auditorIdx}} \bmod \rsaMod$. 

\subsection{Realizing \contentModFuncExact}
%
A randomizable set homomorphic signature \ourSigScheme almost immediately provides a protocol realizing $\contentModFuncExact$ (outlined in \figref{fig:sig_use}). We assume that the client wants to upload files that are drawn from the universe \universe, and the content hash function $\contentHash: \universe \rightarrow \bin^{\genericVecLen}$ maps any such content to an \genericVecLen-bit string. Two items $\genericUniverseElmt, \genericUniverseElmtAlt$ are similar iff. $\contentHash(\genericUniverseElmt) = \contentHash(\genericUniverseElmtAlt)$.  



At a high-level, the protocol has the following steps

\itemListStart
\item \textbf{Creating the blocklist:} The auditors sign the blocklist \blocklist of content hashes and publish as follows
\enumListStart 
\item The auditors jointly sample a key pair for  by invoking \dssFunc, $\langle \auditorPubKeyOurScheme, \auditorPrivKeyOurScheme \rangle$, where the private key is secret-shared
\item Auditor $\auditor[\auditorIdx]$ samples a key pair for a standard digital signature scheme $\langle \auditorPubKeyGenericScheme[\auditorIdx], \auditorPrivKeyGenericScheme[\auditorIdx] \rangle$
\item The auditors invoke $\dssFunc$ with their own shares of the private key to sign the blocklist of content hashes $\genericSig[\blocklist] \assign \signingAlg[\privKey](\blocklist)$. The blocklist verification token $\blklistToken \assign \langle \auditorPubKeyOurScheme, \genericSig[\blocklist] \rangle$
\item The auditors sign $\blklistToken$ under their key own key $ \auditorPrivKeyGenericScheme[\auditorIdx]$ and publish. 
\item The client(s) downloads $\blklistToken$ and verifies the signature under all the auditors' public keys $\{\auditorPubKeyGenericScheme[\auditorIdx]\}_{\auditorIdx \in [1, \numAuditors]}$. 
\enumListEnd
\item \textbf{Uploading content} When uploading content $\clientData$, the client implements the following steps
\enumListStart
\item The client samples a symmetric encryption $\genericSymmKey$ and encrypts the content $\encrypt[\genericSymmKey](\clientData)$
\item The client derives $\genericCInputElmt \assign \contentHash(\clientData)$, and computes 

\begin{align*}
&\genericSig[\blocklist \setminus \{\genericCInputElmt\}] \assign \genericSig[\blocklist] \sigMinus[\auditorPubKeyOurScheme] \{ \genericCInputElmt \} \\
&\randomize[\auditorPubKeyOurScheme](\genericSig[\blocklist \setminus \{\genericCInputElmt \}]) = \langle \genericSigPair[1], \genericSigPair[2] \rangle \\
&\clientMeta \assign \langle \genericSigPair[1], \genericSigPair[2] \oplus \genericSymmKey \rangle
\end{align*}

The client sends $\encrypt[\genericSymmKey](\clientData)$ and $\clientMeta$ to the server

\item The server computes $\verifyRand[\auditorPubKeyOurScheme](\genericSigPair[1])$. If the output is $\genericSigPair[2]$, then output $\genericSigPair[2] \oplus (\genericSigPair[2] \oplus \genericSymmKey) = \genericSymmKey$, and decrypt $\encrypt[\genericSymmKey](\clientData)$ and abort. Otherwise, store $\encrypt[\genericSymmKey](\clientData)$ and send confirmation. 
\enumListEnd



\itemListEnd
